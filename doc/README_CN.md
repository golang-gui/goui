# goui

goui 是一个正在开发中的 Go 跨平台桌面 GUI 框架。

项目希望在不引入第三方原生运行时、不依赖 CGO 编译工具链的前提下，充分利用各操作系统已经提供的窗口、绘图与文字排版能力，并逐步建立一套适合 Go、也适合大模型理解和驱动的现代 GUI 运行时。

> [!IMPORTANT]
> goui 目前仍处于早期开发阶段。平台绘图与文字排版基础已经可用，但高层 UI、稳定 API 和完整事件系统尚未完成，暂不适合生产环境。

## 为什么是 Go

Go 并不拥有成熟、统一的桌面 GUI 生态，但它具备一些很适合构建 GUI 框架的特征：

- 语法和工程模型简单，框架及应用代码容易阅读、维护和生成；
- 原生编译与单文件分发友好，部署时不需要额外语言运行时；
- GC 停顿低，能够兼顾交互延迟与开发效率；
- goroutine 适合承载 I/O、计算和业务任务，UI 状态则可以在事件循环中串行提交；
- 完整的标准工具链适合测试、分析、交叉构建和自动化。

goui 不打算复制某个已有框架，也不试图用 Go 重写整套系统图形栈。项目会把窗口管理、硬件加速绘图、字体塑形（shaping）和文字排版等重型工作交给成熟的系统能力，在其上构建 Go 原生的 UI 抽象。

## 核心目标

### 无 CGO 工具链

goui 通过 [`purego`](https://github.com/ebitengine/purego) 和 [`goexlib/cgo`](https://github.com/goexlib/cgo) 在运行时调用系统 API。这里的 `goexlib/cgo` 是动态调用桥接库，并不要求启用 Go 的 CGO 编译模式。

当前模块要求 Go 1.24。以整个仓库为例，目标构建方式是：

```bash
CGO_ENABLED=0 go build ./...
```

应用构建不需要 C/C++ 编译器、原生头文件或链接脚本。

### 不携带第三方动态库

除目标操作系统提供的系统库外，goui 不要求应用分发额外的第三方动态库。Go 模块依赖参与 Go 构建，不作为额外原生运行时随应用部署。

Linux 环境仍需提供 X11、OpenGL、Pango、Cairo、Fontconfig、GLib 等系统共享库；精简发行版或容器环境可能需要通过系统包管理器安装这些运行时依赖。

### 使用平台擅长的后端

goui 不强求所有平台得到逐像素一致的结果，而是追求布局、结构和交互体验高度相似，同时尊重各平台的字体、栅格化和窗口系统特性。

| 平台 | 窗口系统 | 主要绘图后端 | 文字排版 |
| --- | --- | --- | --- |
| Windows | Win32 | Direct2D | DirectWrite |
| macOS | Cocoa | nanovgo + OpenGL | Core Text |
| Linux | X11 | nanovgo + OpenGL | Pango |

Windows 在 Direct2D 不可用时、macOS 和 Linux 在 OpenGL 初始化失败时，目前还提供软件绘制回退。

### 面向自动化与大模型

可检查性和可驱动性不会作为事后补充。基础 UI 元素从设计之初就需要支持：

- 导出稳定的文本或结构化描述，包括层级、标识、文本、状态、位置和可执行动作；
- 模拟鼠标、键盘、文本输入、焦点切换等事件；
- 查询 UI 状态并进行等待、断言和回放；
- 在此基础上形成可复用的 skill，使大模型能够编写、运行和调试 goui 应用。

这套能力也会直接服务于普通的自动化测试、无障碍语义和调试工具。

## 架构方向

```text
应用与高层 UI
       │
       ├── 命令式控件 API（第一阶段）
       └── 声明式 View / Element 运行时（后续）
       │
布局、输入、焦点、语义与场景图
       │
统一 Graphics / Typography 接口
       │
Platform：窗口、事件循环、系统 API 动态绑定
       │
┌──────────────┬────────────────┬────────────────┐
│ Windows      │ macOS          │ Linux          │
│ Win32        │ Cocoa          │ X11            │
│ Direct2D     │ OpenGL/nanovgo │ OpenGL/nanovgo │
│ DirectWrite  │ Core Text      │ Pango          │
└──────────────┴────────────────┴────────────────┘
```

`platform` 层只负责平台相关能力及绑定，不承担高层控件语义。绘图与文字排版分别通过 `platform/graphics` 和 `platform/typography` 提供统一接口，上层不需要直接依赖 Win32、Cocoa、X11、DirectWrite 或 Pango 类型。

事件处理将采用统一的事件循环抽象，并遵守各平台对 UI 线程的要求。耗时工作可以在 goroutine 中执行，但 UI 树更新、布局、事件分发和帧提交应由 UI 事件循环串行协调。

## 当前进展

已经完成或具备可用基础：

- Windows、macOS 和 Linux 的基础平台绑定；
- Win32、Cocoa 和 X11 窗口实现；
- Direct2D 与 nanovgo/OpenGL 绘图后端；
- DirectWrite、Core Text 与 Pango 文字布局后端；
- 位图、画刷、路径、裁剪、基础图形、图片和文字绘制接口；
- 多平台绘图及文字排版测试。

正在调整：

- 当前 `EventQueue` 在不同平台上的语义并不完全一致，尤其无法自然匹配 macOS 的应用事件模型；
- 事件系统将重构为事件循环抽象，再向上承载输入分发、UI 调度和帧生命周期；
- 现有公开 API 仍可能发生较大变化。

尚未实现：

- 可供应用使用的完整高层 UI API；
- 基础控件、布局、焦点、键盘输入与 IME；
- 场景图、声明式 UI 和稳定的状态更新模型；
- 面向大模型和自动化测试的完整检查、模拟与回放协议。

## 路线图

### 1. 事件循环与命令式基础 UI

- 将平台事件队列调整为统一的事件循环抽象；
- 建立窗口、UI 调度、输入分发、失效和重绘的最小闭环；
- 实现 `Label`、`Button`、`TextInput`、`ImageView` 和 `ListView`；
- 实现 `BoxLayout` 与 `FlexBoxLayout`；
- 用小而完整的示例快速验证 API、生命周期、布局和跨平台行为。

第一阶段优先验证核心设计，不追求一次性建立庞大的控件库。

### 2. 可描述、可模拟、可调试

- 为 UI 元素定义稳定身份、语义信息和结构化描述；
- 建立统一的指针、键盘、文本和焦点事件模型；
- 支持输入模拟、UI 查询、等待、断言和事件回放；
- 提供适合 skill 调用的应用检查与驱动接口；
- 让大模型能够基于实际运行状态定位问题，而不仅依赖源码和截图猜测。

这些要求会与基础控件同步设计，而不是等控件体系完成后再改造。

### 3. 场景图与声明式 UI

- 将绘制结果表达为可检查、可缓存的场景图；
- 分离声明式描述、持久化元素、布局对象和绘制节点；
- 实现状态驱动的 UI 构建、更新与局部失效；
- 支持更精确的重绘、缓存、命中测试和帧调度；
- 在不牺牲 Go 可读性的前提下，形成真正现代化的声明式 GUI 框架。

## 设计原则

- **平台原生能力优先**：复用成熟的系统绘图、字体和窗口设施。
- **大致一致而非像素一致**：统一框架语义，保留平台最佳表现。
- **Go 工具链优先**：不要求 CGO、C 编译器和额外原生构建流程。
- **显式的平台边界**：平台差异集中在 `platform`，不向控件层扩散。
- **事件循环拥有 UI 状态**：后台 goroutine 不直接并发修改 UI 树。
- **可测试性是一等能力**：UI 描述、事件模拟和状态检查属于核心协议。
- **先验证再扩展**：先用最小控件集验证设计，再建设声明式运行时和完整生态。

## 仓库结构

```text
core/
  geometry/              基础几何类型

platform/
  common/                平台公共接口
  events/                当前事件类型
  graphics/              统一绘图接口及各绘图后端
  typography/            统一文字排版接口及各平台实现
  windows/               Win32 与 Windows SDK 动态绑定
  darwin/                Cocoa 与 macOS Framework 动态绑定
  linux/                 X11 及 Linux 系统库动态绑定
```

## 参与开发

当前阶段最重要的是稳定平台事件循环并完成第一组命令式控件。提交改动时，请优先保证：

- `CGO_ENABLED=0` 的构建目标不被破坏；
- 不新增需要随应用分发的第三方动态库；
- 平台差异被限制在清晰的后端边界；
- 新 UI 能力同时考虑可描述性和输入模拟；
- 不以牺牲平台文字与交互质量来换取表面的像素一致。
